2. Propósito del TP

El objetivo de este trabajo práctico es evaluar la capacidad del alumnado para integrar de manera lógica todos los conceptos abordados en las clases.:

  • Creación de bases de datos y modelado de tablas.
  • Asignación adecuada de todos los tipos de datos enseñados.
  • Aplicación estricta de restricciones estructurales: PRIMARY KEY, FOREIGN KEY, NOT NULL, UNIQUE y DEFAULT.
  • Construcción de filtros avanzados y ordenamientos (WHERE, ORDER BY, DISTINCT, IS NULL, IS NOT NULL).
  • Relacionamiento de tablas mediante INNER JOIN, LEFT JOIN y RIGHT JOIN.
  • Manejo de integridad referencial con ON UPDATE CASCADE y ON DELETE CASCADE.
  • Agrupación de datos y uso de funciones específicas: YEAR(), MONTH(), DAY().

3. Preparación inicial del script

Al inicio de su archivo SQL, deben plantear la estructura base para que el script pueda ser ejecutado múltiples veces sin generar errores de duplicación. Deben incluir:

  • Comandos DROP DATABASE IF EXISTS o DROP TABLE IF EXISTS según corresponda.
  • Creación de la base de datos utilizando CREATE DATABASE IF NOT EXISTS y tablasCREATE TABLE IF NOT EXISTS.
  • Selección de la base de datos con USE.
  • A modo de apoyo y comprobación, incluyan los comandos SHOW DATABASES; y SHOW TABLES; (pueden dejarlos comentados).
  • Aseguren una nomenclatura coherente y estandarizada para todos sus objetos (por ejemplo, uso consistente de minúsculas y guiones bajos).

4. Parte A: Relación 1 a 1 y Modificaciones

  1. Crear una tabla principal con la temática escogida que contenga al menos 6 registros. La columna que actúe como clave primaria debe nombrarse estrictamente siguiendo este formato: ID_nombre_de_tu_tabla.
  2. Crear una segunda tabla diseñada para tener una relación 1 a 1 con la primera. No olviden definir la columna que servirá como clave foránea (FK) y ON DELETE CASCADE ,ON UPDATE CASCADEpara relacionarlas.
  3. Entre ambas tablas, es obligatorio aplicar todos los tipos de datos enseñados hasta el momento (por ejemplo: INT, VARCHAR, DECIMAL, DATE, BOOLEAN, TEXT, etc.).
  4. A las columnas que consideren pertinentes, apliquen las restricciones DEFAULT, NOT NULL y UNIQUE.
  5. Insertar al menos 4 registros en la segunda tabla y relacionarlos lógicamente con los registros de la primera tabla.
  6. Ejercicios con valores NULL y DEFAULT:
    • Insertar un registro dejando explícitamente una columna como NULL. Luego, actualizar dicho registro asignándole un dato válido.
    • Insertar un registro permitiendo que actúe el valor DEFAULT en una columna. Posteriormente, actualizar ese registro modificando el valor por defecto.
  7. Modificación de estructuras (ALTER):
    • Cambiar el nombre de una columna existente.
    • Modificar el tipo de dato de una columna existente.
    • Agregar una nueva columna con el tipo de dato que deseen en cada una de las dos tablas.
  8. Consultas con alias y tipos de JOIN:
    • Crear un filtro que muestre registros relacionados utilizando alias, basado en dos columnas de la tabla padre y una de la tabla hija.
    • Crear un filtro que muestre registros relacionados utilizando alias, basado en una columna de la tabla padre y dos de la tabla hija (eviten repetir las columnas del ítem anterior).
    • Crear un filtro que muestre registros relacionados utilizando alias, basado en tres columnas de la tabla padre y tres de la tabla hija.
  9. Prácticas específicas:
    • Actualizar uno o varios datos en una columna del tipo DECIMAL. Crear un filtro para esta columna aplicando LIMIT.
    • Actualizar uno o varios datos en una columna del tipo DATE. Filtrar la tabla utilizando las funciones YEAR(), MONTH() o DAY().
    • Eliminar al menos un registro de la tabla padre.

5. Parte B: Relación 1 a Muchos (1:N)

Amplíen su base de datos creando un escenario que represente una relación de "uno a muchos" (por ejemplo: un cliente realiza muchas compras, un autor escribe muchos libros).

  • Definir claramente las tablas con sus respectivas PRIMARY KEY y FOREIGN KEY.
  • La tabla padre (lado "1") debe contener al menos 4 registros.
  • La tabla hija (lado "muchos") debe contener al menos 8 registros vinculados.
  • Redactar al menos 4 consultas demostrativas:
    1. Una consulta que muestre todos los registros relacionados (padre e hijos).
    2. Una consulta que aplique un filtro sobre un dato de la tabla padre.
    3. Una consulta que aplique un filtro sobre un dato de la tabla hija.
    4. Una consulta ordenada utilizando ORDER BY.
  • Dentro de las consultas anteriores, es obligatorio utilizar al menos un INNER JOIN y un LEFT JOIN o RIGHT JOIN.
  • Realizar un DELETE sobre un registro de la tabla padre y que borre los datos relacionados en las otras tablas.

6. Parte C: Relación Muchos a Muchos (N:M)

Implementar un escenario de relación "muchos a muchos" mediante una tabla intermedia o pivote.

  • Crear dos tablas principales y una tabla intermedia.
  • Volumen de datos mínimo: insertar 4 registros en la primera tabla principal, 4 registros en la segunda, y al menos 7 registros en la tabla intermedia.
  • La tabla intermedia debe contener, además de las claves foráneas, al menos un dato propio adicional (por ejemplo: fecha_registro, estado, calificación, etc.).
  • Redactar consultas que empleen JOIN y el uso de alias (AS) para:
    • Mostrar todos los elementos de la tabla B asociados a un registro específico de la tabla A.
    • Mostrar todos los elementos de la tabla A asociados a un registro específico de la tabla B.
  • Diseñar dos consultas que aplique la cláusula ORDER BY.
